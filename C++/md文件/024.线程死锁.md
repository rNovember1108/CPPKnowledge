# 线程死锁

## 锁的粒度

锁的粒度（Lock Granularity）是并发控制中衡量锁作用范围的核心概念，指锁保护的数据单元大小。其选择直接影响系统并发度和性能开销，需根据具体场景权衡‌。

### 定义与分类

- **粗粒度锁**‌ 
  锁定范围大（如整个表、类或方法），并发度低但管理开销小。例如表级锁（Table Lock）会阻塞整张表的访问‌。

- **细粒度锁**‌ 
  锁定范围小（如行、字段或数据项），并发度高但管理复杂。例如行级锁（Row Lock）仅锁定单行记录‌。

- ‌**中间粒度锁**‌
  如页级锁（Page Lock）锁定数据页（如8KB），平衡并发与开销‌。

### 性能影响

- ‌**并发度**‌：粒度越小，并发访问能力越强（细粒度锁允许多线程同时操作不同数据）‌。
- ‌**开销**‌：细粒度锁需更多资源管理（如锁表、死锁检测），粗粒度锁则反之‌。
- ‌**竞争概率**‌：粗粒度锁易引发线程竞争，细粒度锁通过分散锁降低竞争‌。

### 选择建议

- ‌**高并发场景**‌：优先细粒度锁（如行锁、字段锁）‌。
- ‌**低开销需求**‌：选择粗粒度锁（如类锁、表锁）‌。
- ‌**混合策略**‌：结合意向锁（如IS/IX锁）实现多粒度协同‌。

## 死锁的定义

在C++中，死锁（Deadlock）是指两个或两个以上的执行单元（进程、线程）在执行过程中，因为竞争资源而陷入了互相等待的状态，导致没有一个执行单元能够继续执行。死锁通常发生在多执行单元环境中，当多个执行单元同时锁定资源而又试图获取其他执行单元已锁定的资源时，就可能发生死锁。

## 死锁的产生条件

产生死锁四个必要条件：

- 互斥条件：指执行单元对所分配到的资源进行排它性使用，即在一段时间内某资源只由一个执行单元占用。如果此时还有其它执行单元请求资源，则请求者只能等待，直至占有资源的执行单元用毕释放。

- 请求和保持条件：指执行单元已经保持至少一个资源，但又提出了新的资源请求，而该资源已被其它执行单元占有，此时请求执行单元阻塞，但又对自己已获得的资源保持不放。

- 非剥夺条件：指执行单元已获得的资源，在未使用完之前，不能被剥夺，只能在使用完时由自己释放。

- 循环等待：指在发生死锁时，必然存在一个执行单元——资源的环形链，即执行单元集合{P0，P1，P2，···，Pn}中的P0正在等待一个P1占用的资源；P1正在等待P2占用的资源，……，Pn正在等待已被P0占用的资源。

## 死锁的产生场景

### 线程对同一个锁反复加锁

如果同一线程对同一个锁加锁两次，那么这会导致死锁，这种情况在使用自释放锁时经常出现，比如 c++ 11 的 `lock_guard<mutex>` ，如果在 `lock_guard` 还没释放时再次对同一个锁使用 `lock_guard<mutex>`，则会立刻导致死锁，造成程序异常。

**代码示例如下**：

```
std::mutex mutex1;
void ThreadTest()
{
    std::lock_guard<std::mutex> lock1(mutex1);
    std::this_thread::sleep_for(std::chrono::milliseconds(1));
    std::lock_guard<std::mutex> lock2(mutex1);
    std::cout << "Thread success\n";
}
```

**解决办法**：

使用递归锁`std::recursive_mutex` ，通过‌**锁计数器**‌实现可重入性。其工作原理如下：

- **锁计数器机制**‌ 
  同一线程每次调用`lock()`时，内部计数器递增；调用`unlock()`时计数器递减。仅当计数器归零时，锁才真正释放供其他线程获取。

- ‌**与线程ID的关联**‌ 
  虽然实现上会记录持有锁的线程ID（用于验证重入权限），但避免死锁的关键在于允许同一线程多次加锁，而非单纯依赖ID检测。若线程A持有锁时再次请求，递归锁会通过计数器放行，而普通`std::mutex`会因重复加锁直接死锁。

- **性能权衡**‌ 
  递归锁因维护计数器及线程ID信息，性能略低于普通互斥锁，适用于递归调用或复杂回调等必须嵌套加锁的场景。

```
std::recursive_mutex mutex1;
void ThreadTest()
{
    std::lock_guard<std::recursive_mutex> lock1(mutex1);
    std::this_thread::sleep_for(std::chrono::milliseconds(1));
    std::lock_guard<std::recursive_mutex> lock2(mutex1);
    std::cout << "Thread success\n";
}
```

### 多线程 + 多锁

如果同时存在多个线程和多个锁，而每个线程都有机会操作所有的锁，那么这就可能导致死锁,比如线程1加锁A，等待加锁B，而线程2加锁B，等待加锁A。

**代码示例如下**：

    #include <iostream>
    #include <thread>
    #include <mutex>
    
    std::mutex mutex1, mutex2;
    
    void ThreadA() 
    {
        std::lock_guard<std::mutex> lock1(mutex1);
        std::this_thread::sleep_for(std::chrono::milliseconds(1)); // 让ThreadB有机会锁定mutex2
        std::lock_guard<std::mutex> lock2(mutex2);
        std::cout << "Thread A\n";
    }
    
    void ThreadB() 
    {
        std::lock_guard<std::mutex> lock2(mutex2);
        std::this_thread::sleep_for(std::chrono::milliseconds(1)); // 让ThreadA有机会锁定mutex1
        std::lock_guard<std::mutex> lock1(mutex1);
        std::cout << "Thread B\n";
    }
    
    int main() 
    {
        std::thread t1(ThreadA);
        std::thread t2(ThreadB);
        t1.join();
        t2.join();
        return 0;
    }

在这个代码中，`ThreadA` 首先尝试锁定`mutex1`然后是`mutex2`，而`ThreadB`则是首先锁定`mutex2`然后是`mutex1`。如果两个线程几乎同时运行，它们可能会各自锁定一个互斥量，然后永远等待另一个互斥量的释放，这就导致了死锁。

**解决办法一**：

始终按照一致的顺序对多个锁进行加锁，比如有两个锁 A 和 B，那么任何线程在对 A 和 B加锁时都要按照 “先A后B” 或者 “先B后A” 的顺序进行即可。

```
void ThreadA()
{
    std::lock_guard<std::mutex> lock1(mutex1);
    std::this_thread::sleep_for(std::chrono::milliseconds(1)); // 让ThreadB有机会锁定mutex2
    std::lock_guard<std::mutex> lock2(mutex2);
    std::cout << "Thread A\n";
}

void ThreadB()
{
    std::lock_guard<std::mutex> lock1(mutex1);
    std::this_thread::sleep_for(std::chrono::milliseconds(1)); // 让ThreadA有机会锁定mutex1
    std::lock_guard<std::mutex> lock2(mutex2);
    std::cout << "Thread B\n";
}
```

**解决方法二**：

通过非阻塞的`try_lock()`尝试获取锁，若失败则释放已持有的锁并重试，打破循环等待条件。这种方式相比直接`lock()`能有效预防死锁，但需注意活锁风险（线程持续重试）以及权衡频繁重试带来的性能开销。

```
void ThreadA()
{
    while(true)
    {
        if(mutex1.try_lock())
        {
            std::this_thread::sleep_for(std::chrono::milliseconds(1));
            if(mutex2.try_lock())
            {
                std::cout << "Thread A\n";
                mutex2.unlock();
                mutex1.unlock();
                break;
            }
            else
            {
                mutex1.unlock();
                std::this_thread::sleep_for(std::chrono::milliseconds(1));
            }
        }
    }
}

void ThreadB()
{
    while(true)
    {
        if(mutex2.try_lock())
        {
            std::this_thread::sleep_for(std::chrono::milliseconds(1));
            if(mutex1.try_lock())
            {
                std::cout << "Thread B\n";
                mutex1.unlock();
                mutex2.unlock();
                break;
            }
            else
            {
                mutex2.unlock();
                std::this_thread::sleep_for(std::chrono::milliseconds(1));
            }
        }
    }
}
```

**解决方法三：**

倘若实在无法确保顺序，那么可以使用c++ 11提供的 `std::scoped_lock` （参见上一章21.1.2.4）进行同步加锁，此组件可以保证传入的多个锁可以在不死锁的情况下依次锁住。

```
void ThreadA()
{
    std::scoped_lock<std::mutex, std::mutex> lock1(mutex1, mutex2);
    std::this_thread::sleep_for(std::chrono::milliseconds(1)); // 让ThreadB有机会锁定mutex2
    std::cout << "Thread A\n";
}

void ThreadB()
{
    std::scoped_lock<std::mutex, std::mutex> lock1(mutex1, mutex2);
    std::this_thread::sleep_for(std::chrono::milliseconds(1)); // 让ThreadA有机会锁定mutex1
    std::cout << "Thread B\n";
}
```

**解决方法四：**

使用作用域或者函数封装等利用`RAII`的方式让智能锁（如`std::lock_guard`或`std::unique_lock`）的作用域限制在代码块或函数内，避免死锁。

```
void ThreadA()
{
    {
        std::lock_guard<std::mutex> lock1(mutex1);
    }

    std::this_thread::sleep_for(std::chrono::milliseconds(1)); // 让ThreadB有机会锁定mutex2

    {
        std::lock_guard<std::mutex> lock2(mutex2);
    }

    std::cout << "Thread A\n";
}

void ThreadB()
{
    {
        std::lock_guard<std::mutex> lock2(mutex2);
    }

    std::this_thread::sleep_for(std::chrono::milliseconds(1)); // 让ThreadA有机会锁定mutex1

    {
        std::lock_guard<std::mutex> lock1(mutex1);
    }

    std::cout << "Thread B\n";
}
```

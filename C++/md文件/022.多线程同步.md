# 多线程同步

## 标准C++多线程同步

### 互斥量(Mutexes)

互斥锁用于保护共享资源，确保同一时间只有一个线程可以访问该资源。

| 互斥量                   | 版本    | 作用                                      |
| --------------------- | ----- | --------------------------------------- |
| mutex                 | C++11 | 最基本的互斥量                                 |
| timed_mutex           | C++11 | 有超时机制的互斥量                               |
| recursive_mutex       | C++11 | 可重入的互斥量                                 |
| recursive_timed_mutex | C++11 | 结合 timed_mutex 和 recursive_mutex 特点的互斥量 |
| shared_timed_mutex    | C++14 | 具有超时机制的可共享互斥量                           |
| shared_mutex          | C++17 | 共享的互斥量                                  |

这个系列的对象均提供了加锁（`lock()`）、尝试加锁（`trylock()`）和解锁（`unlock()`）的方法。为了避免死锁， lock() 和 unlock() 方法需要成对使用。

#### std::mutex(C++ 11)

独占式访问资源，同一时间只可以被一个线程锁定,同一个锁多次锁定会造成死锁；通过成员函数`lock()`进行上锁，unlock进行解锁；

```
std::mutex mtx;
mtx.lock();
// 访问共享资源
mtx.unlock();
```

#### std::recursive_mutex(C++ 11)

这是递归互斥量，允许同一时间多次被同一线程获得这个互斥量而不会死锁，允许同一线程多次锁定，只要解锁的次数和锁定的次数相同，互斥量就会被释放；

```
std::recursive_mutex recursiveMutex;
void recursiveLock(int i) {
 if (i <= 0) return;
 recursiveMutex.lock();
 std::cout << "Locked " << i << " times.\n";
 recursiveLock(i - 1);
 recursiveMutex.unlock();
}
```

#### std::time_mutex(C++ 11)

额外允许传递一个时间段或时间点，用来定义多长时间内它可以尝试捕捉一个`lock`，为此它提供了`try_lock_for()`和`try_lock_until()`

- `try_lock_for(duration)`：尝试锁定互斥量，最多等待指定的时间 `duration`。
- `try_lock_until(time_point)`：尝试锁定互斥量，直到指定的时间点 `time_point`。

```
oid threadFunction() 
{
    if (mtx.try_lock_for(std::chrono::seconds(1))) 
    {
        std::cout << "Locked\n";
        // 进行操作
        std::this_thread::sleep_for(std::chrono::seconds(2));
        mtx.unlock();
    } 
    else 
    {
        std::cout << "Failed to lock\n";
    }
}


void threadFunction() 
{
    auto timeout = std::chrono::steady_clock::now() 
    + std::chrono::seconds(2); // 2秒后的时间点
    if (mtx.try_lock_until(timeout)) 
    {
        std::cout << "Locked\n";
        // 进行操作
        std::this_thread::sleep_for(std::chrono::seconds(1));
        mtx.unlock();
    } 
    else 
    {
        std::cout << "Failed to lock\n";
    }
}
```

#### std::recursive_timed_mutex(C++ 11)

允许同一线程多次取得`lock`，可以指定期限，是`std::recursive_mutex`和`std::time_mutex`的结合。

#### std::shared_mutex (C++ 17)

底层实现主要原理是操作系统提供的读写锁，也就是说，在存在多个线程对共享资源读、少许线程对共享资源写的情况下，`std::shared_mutex `比 `std::mutex` 效率更高。

`std::shared_mutex `提供了` lock() `和` unlock() `方法获取写锁和解除写锁，提供了 `lock_shared() `和` unlock_shared()` 方法获取读锁和解除读锁，写锁模式我们称为排他锁（Exclusive Locking），读锁模式我们称为共享锁（Shared Locking）。

排他性锁定方法有：

| 函数       | 功能描述             | 备注     |
| -------- | ---------------- | ------ |
| lock     | 锁定互斥，若互斥不可用则阻塞   | 公有成员函数 |
| try_lock | 尝试锁定互斥，若互斥不可用则返回 | 公有成员函数 |
| unlock   | 解锁互斥             | 公有成员函数 |

共享锁定方法有：

| 函数              | 功能描述                   | 备注     |
| --------------- | ---------------------- | ------ |
| lock_shared     | 为共享所有权锁定互斥，若互斥不可用则阻塞   | 公有成员函数 |
| try_lock_shared | 尝试为共享所有权锁定互斥，若互斥不可用则返回 | 公有成员函数 |
| unlock_shared   | 解锁互斥（共享所有权）            | 公有成员函数 |

另外，C++ 新标准中引入与 std::shared_mutex 配合使用的两个对象—— **std::unique_lock**和**std::shared_lock** ，这两个对象在构造时自动对`std::shared_mutex`进行加锁、在析构时自动对`std::shared_mutex`进行解锁，前者用于加解 `std::shared_mutex` 的写锁，后者用于加解 `std::shared_mutex `的读锁。

```
std::shared_mutex mtx;
void reader() 
{
    std::shared_lock<std::shared_mutex> lock(mtx);
    std::cout << "Reading sharedData";
}

void writer() 
{
    std::unique_lock<std::shared_mutex> lock(mtx);
    std::cout << "Writing sharedData";
}
```

#### 读写锁的写饥饿

**形成原因**

1. 读线程高并发，当读线程频繁获取共享锁且执行时间较长时，写线程可能因无法获取独占锁而无限等待。例如高频读取配置的场景中，若读操作未及时释放锁，写操作会被阻塞。

#### std::shared_timed_mutex(C++ 14)

`shared_timed_mutex`类拥有`shared_mutex`一样特性：拥有二个层次的访问：

- **共享** --- 多个线程能共享同一互斥的所有权。其对应的就是读的访问权限。
- **独占性** --- 仅一个线程能占有互斥。其对应的就是写的访问权限。

但与`shared_mutex`不同的是，其增加了与以类似` timed_mutex`的行为， `shared_timed_mutex` 提供通过 `try_lock_for()` 、`try_lock_until()`、`try_lock_shared_for()` 、`try_lock_shared_until()`方法，试图带时限地要求 `shared_timed_mutex`所有权的能力。

排他性锁定方法有：

| 函数             | 功能描述                      | 备注     |
| -------------- | ------------------------- | ------ |
| lock           | 锁定互斥，若互斥不可用则阻塞            | 公有成员函数 |
| try_lock       | 尝试锁定互斥，若互斥不可用则返回          | 公有成员函数 |
| try_lock_for   | 尝试锁定互斥，若互斥在指定的时限时期中不可用则返回 | 公有成员函数 |
| try_lock_until | 尝试锁定互斥，若直至抵达指定时间点互斥不可用则返回 | 公有成员函数 |
| unlock         | 解锁互斥                      | 公有成员函数 |

共享锁定方法有：

| 函数                    | 功能描述                            | 备注     |
| --------------------- | ------------------------------- | ------ |
| lock_shared           | 为共享所有权锁定互斥，若互斥不可用则阻塞            | 公有成员函数 |
| try_lock_shared       | 尝试为共享所有权锁定互斥，若互斥不可用则返回          | 公有成员函数 |
| try_lock_shared_for   | 尝试为共享所有权锁定互斥，若互斥在指定的时限时期中不可用则返回 | 公有成员函数 |
| try_lock_shared_until | 尝试为共享所有权锁定互斥，若直至抵达指定时间点互斥不可用则返回 | 公有成员函数 |
| unlock_shared         | 解锁互斥（共享所有权）                     | 公有成员函数 |

### 互斥量管理

为了避免死锁， **lock()** 和 **unlock()** 方法需要成对使用，但是如果一个函数中有很多出口，而互斥体对象又是需要在整个函数作用域保护的资源，那么在编码时因为忘记在某个出口处调用 **unlock()** 而造成死锁，推荐使用利用 **RAII 技术**封装这两个接口，其实 C++ 11 标准也想到了整个问题，因为已经为我们提供了如下封装：

| 互斥量管理       | 版本    | 作用          |
| ----------- | ----- | ----------- |
| lock_guard  | C++11 | 基于作用域的互斥量管理 |
| unique_lock | C++11 | 更加灵活的互斥量管理  |
| shared_lock | C++14 | 共享互斥量的管理    |
| scope_lock  | C++17 | 多互斥量避免死锁的管理 |

#### std::lock_guard(C++ 11)

**特点**:

- **基本用法**：简单、轻量级的锁管理。
- **自动解锁**：在作用域结束时自动解锁。
- **构造函数**：只有构造函数和析构函数，不能显式解锁。
- **适用场景**：当需要一个简单的互斥量保护，且不需要更复杂的锁控制时。
- **性能开销小**：由于功能简单，性能开销较小。

**适用场景**:

- 当锁定范围非常明确且简单，如在函数的起始和结束处保护一个共享资源。
- 当不需要在同一作用域内进行手动解锁或重新加锁操作时。

```
std::mutex mtx;
void threadFunction() 
{
    std::lock_guard<std::mutex> lock(mtx);
    std::cout << "In critical section\n";
    // 自动解锁
}
```

#### std::unique_lock(C++ 11)

**特点**:

- **基本用法**：提供更灵活的锁管理功能。
- **自动解锁**：在析构函数中自动解锁。
- **构造函数**：支持延迟锁定、显式解锁和重新锁定。
- **适用场景**：当需要在一个作用域内对锁进行复杂的操作（如条件变量、手动解锁）时使用。
- **性能开销**：由于提供了更多功能，相比 `std::lock_guard` 性能开销略大。

**适用场景**:

- 当需要在同一作用域内手动释放锁，或者需要在不同的作用域中重新加锁时。
- 当需要与 `std::condition_variable` 一起使用以实现线程的等待和通知机制时。
- 当锁定范围不够明确，需要更灵活的锁控制时。

```
std::mutex mtx;
void threadFunction() 
{
    std::unique_lock<std::mutex> lock(mtx);
    std::cout << "In critical section\n";
    std::this_thread::sleep_for(std::chrono::seconds(1));
    // 可以显式解锁
    lock.unlock();
    // 之后可以重新锁定
    lock.lock();
}
```

#### std::shared_lock(C++ 14)

C++14 标准中引入的一个类，用于与 `std::shared_mutex` 配合使用，以实现共享（读）锁的功能。

**特点:**

- **共享锁**：允许多个线程同时持有共享锁来读取数据。
- **构造函数**：可以使用 `std::shared_lock` 来获取 `std::shared_mutex` 的共享锁，或者使用 `std::shared_lock` 的构造函数来管理锁的生命周期。
- **自动释放**：当 `std::shared_lock` 对象被销毁时，它会自动释放持有的共享锁。

**适用场景**:

- 需要读写锁的时候。

示例见上文`std::shared_mutex`的示例。

#### std::scoped_lock(C++ 17)

在 C++17 标准中引入的，它提供了一种简洁且安全的方式来管理多个互斥锁的锁定。`std::scoped_lock` 是一种模板类，可以用于锁定一个或多个 `std::mutex`、`std::recursive_mutex`、`std::shared_mutex` 等互斥量，并在对象销毁时自动释放这些锁。

**特点:**

- **锁定多个互斥量**：可以同时锁定多个互斥量，防止死锁和锁竞争。
- **自动解锁**：当 `std::scoped_lock` 对象被销毁时，它会自动解锁所有已锁定的互斥量。
- **避免死锁**：由于所有互斥量都在构造时同时锁定，可以减少死锁的风险。

```
std::mutex mutex1;
std::mutex mutex2;

void task1()  
{
    std::scoped_lock lock(mutex1, mutex2);
    std::cout << "Task 1 is executing\n";
    // 执行任务
}

void task2()  
{
    std::scoped_lock lock(mutex1, mutex2);
    std::cout << "Task 2 is executing\n";
    // 执行任务
}
```

### 条件变量（Condition Variables）(C++ 11)

`std::condition_variable`：与互斥锁一起使用。

条件变量是 C++11 中提供的一种多线程同步机制，它允许一个或多个线程等待另一个线程发出通知，以便能够有效地进行线程同步。

`std::condition_variable `需要与`mutex`（互斥锁）一起使用。当线程需要等待某个条件变成真时，它会获取一个互斥锁，然后在条件变量上等待，等待期间会自动释放互斥锁。另一个线程在满足条件后会获取相同的互斥锁，并调用条件变量的` notify_one() `或 `notify_all() `函数来唤醒等待的线程。

`std::condition_variable` 有三个等待函数：`wait()`、`wait_for()` 和 `wait_util()`，这三个函数需要与互斥锁一起使用，以互斥的方式访问共享资源，并阻塞线程，等待通知。

#### wait

```
void wait(std::unique_lock<std::mutex>& lock);
template <class Predicate>
void wait (unique_lock<mutex>& lck, Predicate pred);
```

`wait() `函数用于阻塞线程并等待唤醒。

在调用 `wait()` 之前，必须获取一个独占锁（`std::unique_lock`）并将它传递给` wait()` 函数。

如果条件变量当前不满足，线程将被阻塞，同时释放锁，使得其他线程可以继续执行。当另一个线程调用 `notify_one() `或 `notify_all() `来通知条件变量时，被阻塞的线程将被唤醒，并再次尝试获取锁。`wait() `函数返回时，锁会再次被持有。

`wait() `函数有一个带谓词的版本，可以简化对条件的判断。仅仅有当 pred 条件为 false 时调用 `wait() `才会阻塞当前线程，解决了的唤醒丢失问题。而且在收到其它线程的通知后仅仅有当 pred 为 true 时才会被解除堵塞，解决了虚假唤醒的问题。

```
std::condition_variable cv;
std::mutex mtx;
bool condition = false;

void worker()  
{
    std::unique_lock<std::mutex> lock(mtx);
    cv.wait(lock, []{ return condition; }); // 等待条件满足
    // ...
}
```

#### wait_for

```
template <class Rep, class Period>
cv_status wait_for(std::unique_lock<std::mutex>& lock, const std::chrono::duration<Rep, Period>& rel_time);

template <class Rep, class Period, class Predicate>
bool wait_for (unique_lock<mutex>& lck, const chrono::duration<Rep,Period>& rel_time, Predicate pred);
```

`wait_for() `函数用于阻塞线程并等待唤醒，但与` wait() `不同，它可以设置一个超时时间。在调用` wait_for() `之前，必须获取一个独占锁（`std::unique_lock`）并将它传递给 wait_for() 函数。

如果条件变量在指定的超时时间内变为满足，线程将被唤醒，并且 `wait_for()` 返回 `cv_status::no_timeout`。如果超时时间到期且仍未收到唤醒通知，`wait_for() `返回 `cv_status::timeout`，线程继续执行。

`wait_for()` 函数同样有一个谓词版本，用法同 `wait() `函数。

#### wait_until

```
template <class Clock, class Duration>
cv_status wait_until(std::unique_lock<std::mutex>& lock, const std::chrono::time_point<Clock, Duration>& abs_time);

template <class Clock, class Duration, class Predicate>
bool wait_until(unique_lock<mutex>& lck, const chrono::time_point<Clock,Duration>& abs_time, Predicate pred);
```

`wait_until()` 接受一个绝对时间点作为参数。线程将等待直到指定的绝对时间点，如果在该时间点之前条件变量满足，它将返回并继续执行。如果到达指定时间点仍未收到唤醒通知，`wait_until ()`返回` cv_status::timeout`，线程继续执行。

`wait_until() `函数同样有一个谓词版本，用法同 `wait() `函数。

#### notify_one

```
void notify_one() noexcept;
```

`notify_one()` 用于唤醒等待在条件变量上的单个线程。

如果有多个线程在条件变量上等待，只有其中一个线程会被唤醒，具体是哪个线程 C++ 标准并未明确，所以是不确定的。

被唤醒的线程将尝试获取与条件变量关联的互斥锁，一旦成功获取锁，它可以继续执行。

#### notify_all

```
void notify_all() noexcept;
```

`notify_all()` 用于唤醒等待在条件变量上的所有线程。

如果有多个线程在条件变量上等待，所有这些线程都会被唤醒。

唤醒的线程将竞争获取与条件变量关联的互斥锁，然后可以继续执行。

```
std::mutex mtx;
std::condition_variable cv;
bool ready = false;

void waitingThread()  
{
    std::unique_lock<std::mutex> lock(mtx);
    cv.wait(lock, [] { return ready; });
    std::cout << "Thread resumed\n";
}

void notifyingThread()  
{
    std::this_thread::sleep_for(std::chrono::seconds(1));
    {
        std::lock_guard<std::mutex> lock(mtx);
        ready = true;
    }
    cv.notify_one();
}
```

### 信号量（Semaphores）(C++ 20)

信号量（Semaphore）是一种同步机制，通过维护一个计数来控制多个线程对共享资源的访问。信号量的计数值是一个非负整数，表示可用资源的数量。当线程或进程需要访问共享资源时，它会尝试获取信号量。如果信号量的计数值大于0，则将其减1并允许线程或进程继续执行；如果信号量的计数值为0，则线程或进程将被阻塞，直到信号量的计数值大于0。这样的信号量也称为计数信号量。

除了计数信号量之外，还有二值信号量。二值信号量只有两个状态（0 和 1），类似于互斥锁，主要用于控制单个资源的访问。计数值为1表示资源可用，计数值为0表示资源被占用。

信号量可以用于线程间的同步，使得某些操作按特定顺序执行，从而避免竞态条件的发生，也可以限制对共享资源的并发访问数量，以防止资源过载。

C++20 标准引入了 std::counting_semaphore 和 std::binary_semaphore 类，分别用于实现计数信号量和二值信号量的功能。

信号量的优势如下：

精细控制：信号量可以配置为允许特定数量的线程同时访问资源，实现资源的精细控制。
通用性：信号量更加灵活多样，可以用来实现其他同步原语。
多资源管理：计数信号量可以用来管理多个资源实例，适用于需要控制对资源池（如线程池或连接池）访问的场景。
阻塞等待：信号量中的阻塞和等待机制允许线程等待直到资源再次可用。
超时处理：在获取信号量时可以指定超时，使其更加实用。

信号量对象的主要成员函数包括：

`acquire()`：用于请求信号量。如果计数值大于 0（表示有资源空闲） ，计数值减 1 并立即返回，线程获得对资源的访问权；如果计数值为 0（表示资源都已被占用），则阻塞线程直到信号量变为可用状态（即计数值大于0）。
`release()`：用于释放信号量，增加计数值。

`try_acquire()`：尝试获取信号量而不阻塞。

try_acquire_for() : 设置一个超时时间。

try_acquire_until()：接受一个绝对时间点作为参数。

```
sem.acquire();
// 关键段代码
sem.release();
```

```
if (sem.try_acquire()) 
{
    // 成功获取了信号量
    // 关键段代码
    sem.release();
} 
else 
{
    // 信号量未被获取
});
```

```
if (sem.try_acquire_for(std::chrono::seconds(1))) 
{
    // 在1秒内成功获取了信号量
    // 关键段代码
    sem.release();
} 
else 
{
    // 在1秒内未能获取信号量
}
```

#### 计数信号量(std::counting_semaphore)

计数信号量对象可以通过`std::counting_semaphore`<最大计数值> 信号量对象(初始计数值) 创建，并指定其最大计数值和初始计数值。

```
 std::counting_semaphore<10> sem(1);
```

你可以用一个初始计数来初始化计数信号量，该计数代表可以同时无阻塞访问资源的线程数量。线程可以获取和释放计数，信号量的计数相应地增加或减少。如果线程尝试获取的计数超过了可用的数量，它将阻塞，直到计数变得可用。

```
// 用3个计数初始化信号量
std::counting_semaphore<10> semaphore(3);

void worker(int id)
{
    // 获取
    semaphore.acquire();

    // 执行一些工作
    cout << "Thread " << id << " acquired the semaphore."
         << endl;

    // 释放
    semaphore.release();
    cout << "Thread " << id << " released the semaphore."
         << endl;
}
```

#### 二进制信号量(std::binary_semaphore)

二进制信号量是一种更简单的信号量版本，它只能有两个值：0和1。

通常用于两个线程之间的基本互斥或信号传递。可以被视为具有更轻量级接口的互斥锁。类似于mutex。

```
// 用1个计数（二进制）初始化信号量
binary_semaphore semaphore(1);
void worker(int id)
{
    // 获取信号量
    semaphore.acquire();
    cout << "Thread " << id << " acquired the semaphore."
         << endl;

    // 执行一些工作
    semaphore.release();
    // 释放信号量
    cout << "Thread " << id << " released the semaphore."
         << endl;
}
```

**生产消费者问题**

include <iostream>

### 原子操作（Atomic Operations）(C++ 11)

#### 什么是原子操作

原子操作是指在多线程环境下，不可分割的操作，即在进行操作时不会被其他线程打断。原子操作是一种以单个事务来执行的操作，其他线程只能看到操作完成前或者完成后的资源状态，不存在中间状态可视。从底层来看，原子操作是一些硬件指令，其原子性由硬件保证，C++11 对原子操作进行了抽象，提供了统一的接口，避免使用时嵌入平台相关的代码来支持跨平台使用。

#### 为什么使用原子操作

虽然 C++11 提供了其他高并发的 API，例如互斥锁（`std::mutex`）和条件变量（`std::condition_variable`），但这些 API 在某些情况下并不是最佳选择：

**性能开销**：互斥锁的实现通常依赖于操作系统的系统调用，这可能带来较大的性能开销。相反，`std::atomic<> `的操作通常是无锁的，直接使用底层硬件提供的原子操作，性能更高。
**避免死锁**：使用互斥锁需要小心处理锁的获取和释放，否则容易导致死锁。`std::atomic<>` 不需要显式加锁和解锁，减少了编程复杂性。
**细粒度控制**：`std::atomic<>` 提供了对单个变量的细粒度控制，使得某些简单的并发场景可以更高效地实现。

#### 使用原子操作

`std::atomic<> `是一个模板类，可以实例化各种类型的原子操作。然而，并不是所有类型都可以实例化` std::atomic<> `模板。按照标准的说法，需要是 Trivially Copyable 的类型，简单来说就是满足以下三个条件：

- 连续的内存；

- 拷贝对象意味着按 bit 拷贝（memcpy）；

- 没有虚函数。

用代码来表达则是自定义结构满足下面五个条件：

`std::is_trivially_copyable<T>::value`
`std::is_copy_constructible<T>::value`
`std::is_move_constructible<T>::value`
`std::is_copy_assignable<T>::value`
`std::is_move_assignable<T>::value`

```
std::atomic<int> counter(0);
void increment() 
{
    for (int i = 0; i < 1000; ++i) 
    {
        ++counter;
    }
}

int main() 
{
    std::thread t1(increment);
    std::thread t2(increment);

    t1.join();
    t2.join();

    std::cout << "Counter value: " << counter << '\n';

    return 0;
}
```

#### 注意事项

`std::atomic<>` 只能用于可平凡复制的类型，例如基本数据类型（如 int、float）和某些自定义类型，但` std::string` 不符合这一要求。`std::string` 是一个复杂的类型，包含动态分配的内存，其复制操作涉及更多的资源管理，不能保证在并发环境下安全地进行无锁操作。
可以使用 `std::atomic<std::shared_ptr<std::string>>` 来处理 `std::string`，这是一种间接方式，确保线程安全。

#### 总结

`std::atomic<>` 提供了一种高效、安全的方式来处理多线程环境中的共享数据。通过利用底层硬件的原子操作指令，可以避免使用互斥锁，从而提高性能，并减少死锁的风险。在编写高并发应用程序时，合理使用` std::atomic<> `可以大大简化代码并提高程序的稳定性和性能。

## Windows C++多线程同步

### 互斥锁（Mutex）

#### 临界区(CRITICAL_SECTION)

- **用途**：用于同一进程内的线程同步。
- **特点**：轻量级，比 `HANDLE` 类型的互斥对象效率高。
- **主要函数**：
  - `InitializeCriticalSection(CRITICAL_SECTION* cs)`: 初始化临界区对象。
  - `EnterCriticalSection(CRITICAL_SECTION* cs)`: 进入临界区，若临界区已被其他线程占用，则等待。
  - `LeaveCriticalSection(CRITICAL_SECTION* cs)`: 离开临界区，释放对临界区的占用。
  - `DeleteCriticalSection(CRITICAL_SECTION* cs)`: 删除临界区对象。

**示例代码**：

```
#include <windows.h>
#include <iostream>

CRITICAL_SECTION cs;

void ThreadFunc() {
    EnterCriticalSection(&cs);
    // 临界区代码
    std::cout << "In critical section\n";
    LeaveCriticalSection(&cs);
}

int main() {
    InitializeCriticalSection(&cs);

    HANDLE thread1 = CreateThread(NULL, 0, (LPTHREAD_START_ROUTINE)ThreadFunc, NULL, 0, NULL);
    HANDLE thread2 = CreateThread(NULL, 0, (LPTHREAD_START_ROUTINE)ThreadFunc, NULL, 0, NULL);

    WaitForSingleObject(thread1, INFINITE);
    WaitForSingleObject(thread2, INFINITE);

    DeleteCriticalSection(&cs);

    CloseHandle(thread1);
    CloseHandle(thread2);

    return 0;
}
```

#### HANDLE类型的互斥对象(Mutex)

- **用途**：可以跨进程同步。
- **主要函数**：
  - `CreateMutex(SECURITY_ATTRIBUTES* sa, BOOL initialOwner, LPCTSTR name)`: 创建或打开一个互斥对象。
  - `ReleaseMutex(HANDLE hMutex)`: 释放互斥对象。
  - `WaitForSingleObject(HANDLE hMutex, DWORD milliseconds)`: 等待互斥对象变为可用。
  - `CloseHandle(HANDLE hMutex)`: 关闭互斥对象句柄。

**示例代码**：

```
#include <windows.h>
#include <iostream>

HANDLE hMutex;

DWORD WINAPI ThreadFunc(LPVOID lpParam) {
    WaitForSingleObject(hMutex, INFINITE);
    // 临界区代码
    std::cout << "In critical section\n";
    ReleaseMutex(hMutex);
    return 0;
}

int main() {
    hMutex = CreateMutex(NULL, FALSE, NULL);

    HANDLE thread1 = CreateThread(NULL, 0, ThreadFunc, NULL, 0, NULL);
    HANDLE thread2 = CreateThread(NULL, 0, ThreadFunc, NULL, 0, NULL);

    WaitForSingleObject(thread1, INFINITE);
    WaitForSingleObject(thread2, INFINITE);

    CloseHandle(hMutex);
    CloseHandle(thread1);
    CloseHandle(thread2);

    return 0;
}
```

### 信号量（Semaphore）

- **用途**：用于控制对共享资源的访问，可以设置最大资源数量。
- **主要函数**：
  - `CreateSemaphore(SECURITY_ATTRIBUTES* sa, LONG initialCount, LONG maximumCount, LPCTSTR name)`: 创建或打开一个信号量对象。
  - `ReleaseSemaphore(HANDLE hSemaphore, LONG count, LPLONG previousCount)`: 释放信号量。
  - `WaitForSingleObject(HANDLE hSemaphore, DWORD milliseconds)`: 等待信号量变为可用。
  - `CloseHandle(HANDLE hSemaphore)`: 关闭信号量对象句柄。

**示例代码**：

```
#include <windows.h>
#include <iostream>

HANDLE hSemaphore;

DWORD WINAPI ThreadFunc(LPVOID lpParam) {
    WaitForSingleObject(hSemaphore, INFINITE);
    // 临界区代码
    std::cout << "In critical section\n";
    ReleaseSemaphore(hSemaphore, 1, NULL);
    return 0;
}

int main() {
    hSemaphore = CreateSemaphore(NULL, 1, 1, NULL); // 设置信号量的初始计数和最大计数均为1

    HANDLE thread1 = CreateThread(NULL, 0, ThreadFunc, NULL, 0, NULL);
    HANDLE thread2 = CreateThread(NULL, 0, ThreadFunc, NULL, 0, NULL);

    WaitForSingleObject(thread1, INFINITE);
    WaitForSingleObject(thread2, INFINITE);

    CloseHandle(hSemaphore);
    CloseHandle(thread1);
    CloseHandle(thread2);

    return 0;
}
```

### 事件（Event）

- **用途**：用于线程之间的信号传递。事件可以是手动重置的或自动重置的。
- **主要函数**：
  - `CreateEvent(SECURITY_ATTRIBUTES* sa, BOOL manualReset, BOOL initialState, LPCTSTR name)`: 创建或打开一个事件对象。
  - `SetEvent(HANDLE hEvent)`: 设置事件对象的状态为已信号。
  - `ResetEvent(HANDLE hEvent)`: 将事件对象的状态重置为无信号。
  - `WaitForSingleObject(HANDLE hEvent, DWORD milliseconds)`: 等待事件对象变为已信号。
  - `CloseHandle(HANDLE hEvent)`: 关闭事件对象句柄。

**示例代码**：

```
#include <windows.h>
#include <iostream>

HANDLE hEvent;

DWORD WINAPI ThreadFunc(LPVOID lpParam) {
    WaitForSingleObject(hEvent, INFINITE);
    std::cout << "Event signaled\n";
    return 0;
}

int main() {
    hEvent = CreateEvent(NULL, FALSE, FALSE, NULL); // 自动重置事件，初始状态为无信号

    HANDLE thread = CreateThread(NULL, 0, ThreadFunc, NULL, 0, NULL);

    Sleep(1000); // 等待线程启动
    SetEvent(hEvent); // 设置信号

    WaitForSingleObject(thread, INFINITE);

    CloseHandle(hEvent);
    CloseHandle(thread);

    return 0;
}
```

### 读写锁（SRWLOCK）

- **用途**：允许多个线程同时读取，但仅一个线程可以写入。
- **主要函数**：
  - `InitializeSRWLock(SRWLOCK* srwLock)`: 初始化读写锁。
  - `AcquireSRWLockExclusive(SRWLOCK* srwLock)`: 请求独占访问。
  - `AcquireSRWLockShared(SRWLOCK* srwLock)`: 请求共享访问。
  - `ReleaseSRWLockExclusive(SRWLOCK* srwLock)`: 释放独占访问。
  - `ReleaseSRWLockShared(SRWLOCK* srwLock)`: 释放共享访问。

**示例代码**：

```
#include <windows.h>
#include <iostream>

SRWLOCK srwLock;

void Reader() {
    AcquireSRWLockShared(&srwLock);
    // 读取数据
    std::cout << "Reading\n";
    ReleaseSRWLockShared(&srwLock);
}

void Writer() {
    AcquireSRWLockExclusive(&srwLock);
    // 写入数据
    std::cout << "Writing\n";
    ReleaseSRWLockExclusive(&srwLock);
}

int main() {
    InitializeSRWLock(&srwLock);

    HANDLE readerThread1 = CreateThread(NULL, 0, (LPTHREAD_START_ROUTINE)Reader, NULL, 0, NULL);
    HANDLE readerThread2 = CreateThread(NULL, 0, (LPTHREAD_START_ROUTINE)Reader, NULL, 0, NULL);
    HANDLE writerThread = CreateThread(NULL, 0, (LPTHREAD_START_ROUTINE)Writer, NULL, 0, NULL);

    WaitForSingleObject(readerThread1, INFINITE);
    WaitForSingleObject(readerThread2, INFINITE);
    WaitForSingleObject(writerThread, INFINITE);

    CloseHandle(readerThread1);
    CloseHandle(readerThread2);
    CloseHandle(writerThread);

    return 0;
}
```

### 条件变量（CONDITION_VARIABLE）

- **用途**：用于线程等待特定条件发生。
- **主要函数**：
  - `InitializeConditionVariable(CONDITION_VARIABLE* cv)`: 初始化条件变量。
  - `SleepConditionVariableCS(CONDITION_VARIABLE* cv, CRITICAL_SECTION* cs, DWORD dwMilliseconds)`: 在条件变量上等待。
  - `WakeConditionVariable(CONDITION_VARIABLE* cv)`: 唤醒等待在条件变量上的线程。
  - `WakeAllConditionVariable(CONDITION_VARIABLE* cv)`: 唤醒所有等待在条件变量上的线程。

**示例代码**：

```
include <windows.h>
#include <iostream>

CRITICAL_SECTION cs;
CONDITION_VARIABLE cv;
bool ready = false;

DWORD WINAPI ThreadFunc(LPVOID lpParam) {
    EnterCriticalSection(&cs);
    while (!ready) {
        SleepConditionVariableCS(&cv, &cs, INFINITE);
    }
    // 处理数据
    std::cout << "Condition variable signaled\n";
    LeaveCriticalSection(&cs);
    return 0;
}

int main() {
    InitializeCriticalSection(&cs);
    InitializeConditionVariable(&cv);

    HANDLE thread = CreateThread(NULL, 0, ThreadFunc, NULL, 0, NULL);

    Sleep(1000); // 模拟某些工作

    EnterCriticalSection(&cs);
    ready = true;
    WakeConditionVariable(&cv);
    LeaveCriticalSection(&cs);

    WaitForSingleObject(thread, INFINITE);

    CloseHandle(thread);
    DeleteCriticalSection(&cs);

    return 0;
}
```

## Qt多线程同步

### 信号与槽

Qt 的信号与槽机制可以用来实现线程间的通信。使用 `QThread` 和 `QObject` 的信号与槽机制，能够确保线程安全地传递数据。例如，`QObject::moveToThread()` 将对象移到不同的线程中，并通过信号和槽进行跨线程通信。

### QMutex

`QMutex` 用于确保在同一时间只有一个线程能够访问特定的资源。`QMutexLocker` 是一个方便的 RAII 风格的锁管理工具，能够自动管理锁的获取和释放。

示例代码：

```
QMutex mutex;
QMutexLocker locker(&mutex); // 自动锁定
// 临界区代码
```

### QReadWriteLock

`QReadWriteLock`允许多个线程同时读取共享资源，但在写操作时会排他性地锁住资源。适用于读多写少的场景。

示例代码：

```
QReadWriteLock lock;

void readFunction() {
    QReadLocker locker(&lock); // 读锁
    // 读取共享资源的代码
}

void writeFunction() {
    QWriteLocker locker(&lock); // 写锁
    // 修改共享资源的代码
}
```

### QWaitCondition

`QWaitCondition` 用于在某个条件发生之前让线程等待。通常与 `QMutex` 一起使用来实现条件变量的功能。

示例代码：

```
QMutex mutex;
QWaitCondition condition;
bool ready = false;

void threadFunction() {
    QMutexLocker locker(&mutex);
    while (!ready)
        condition.wait(&mutex);
    // 处理完成后的代码
}
```

### QSemaphore

 `QSemaphore` 用于控制对资源的访问，限制同时访问资源的线程数量。

示例代码：

```
QSemaphore semaphore(3); // 允许同时3个线程访问资源
semaphore.acquire();
// 访问共享资源
semaphore.release();
```

### QAtomicInteger 和 QAtomicPointer

用于进行原子操作，确保在多线程环境中对某些基本类型的操作是线程安全的。

示例代码：

```
QAtomicInteger<int> atomicInt(0); // 初始化为0
atomicInt.fetchAndAdd(1); // 原子地将值增加1
atomicInt.fetchAndSubtract(1); // 原子地将值减少1器
```

### QFutureWatcher

`QFutureWatcher` 用于异步操作的监视和管理，它使得在完成异步操作后能够进行通知。

**关键点**：

- `QFutureWatcher` 可以用于监视 `QFuture` 对象，提供异步操作的完成信号。
- 与 `QtConcurrent::run` 一起使用，可以方便地管理和等待异步任务的结果。

示例代码：

```
QFutureWatcher<void> watcher;

void asyncFunction() {
    QFuture<void> future = QtConcurrent::run([](){
        // 执行异步操作的代码
    });
    watcher.setFuture(future);
}

void onFinished() {
    // 处理异步操作完成后的结果
}

int main() {
    QObject::connect(&watcher, &QFutureWatcher<void>::finished, &onFinished);
    asyncFunction();
    return 0;
}
```

### QEvent

`QEvent` 类提供了事件处理机制，允许不同线程之间通过事件进行通信和同步。具体来说，可以通过以下几种方式利用 `QEvent` 实现线程同步：

#### QCoreApplication::postEvent

`QCoreApplication::postEvent` 方法将事件放入事件队列，并允许线程间通信。它是异步的，因此不会阻塞调用线程。

#### QCoreApplication::sendEvent

`QCoreApplication::sendEvent`适用于需要立即处理事件的场景，通常用于工作线程中同步地处理主线程中的事件，确保事件在当前线程被立即处理，会阻塞调用线程直到事件处理完成。

示例代码：

```
class MyEvent : public QEvent {
public:
    MyEvent() : QEvent(QEvent::User) {}
};

void MyWorkerThreadFunction() {
    QCoreApplication::postEvent(myObject, new MyEvent()); 
    //QCoreApplication::sendEvent(myObject, new MyEvent());
}

// 在主线程中处理事件
bool MyObject::event(QEvent *event) {
    if (event->type() == QEvent::User) {
        // 处理 MyEvent
        return true;
    }
    return QObject::event(event);
}
```

#### QMetaObject::invokeMethod

`QMetaObject::invokeMethod` 可以在特定线程中调用槽函数。配合 `Qt::QueuedConnection` 使用时，确保槽函数在对象所属的线程中执行，这有助于处理线程间同步。

```
void MyWorkerThreadFunction() {
    QMetaObject::invokeMethod(myObject, "mySlot", Qt::QueuedConnection);
}

public slots:
    void mySlot() {
        // 处理来自工作线程的请求
    }
```

## 总结

- 不论是`标准C++`还是`Windows C++`还是`Qt`，他们提供的线程同步方式底层思想都是有相似的，比如**互斥量**、**信号量**、**等待条件**等等方式大家都有。有差别的只是使用时关于API的调用不同。当然也有一些独属于相关框架的方法，比如`Windows C++`的**临界区**、还有Windows和Qt的**事件**等等方法，在相应平台下开发使用更能相得益彰，根据实际情况选择使用最符合的方式。

- 上面有提到**RAII技术**封装，‌‌‌RAII技术（Resource Acquisition Is Initialization）是一种利用‌对象生命周期来控制程序资源的简单技术。最简单的理解就是构造函数加锁，析构函数解锁，利用对象的生命周期来自动实现加解锁，不用担心因为太多的程序控制流程而导致忘了解锁，成为死锁。在使用RAII技术时，当我们在一个很长的函数方法内需要提前解锁，可以使用`{}`作用域的方式。例如：

```
std::mutex mtx;
void threadFunction() 
{
    {
        std::lock_guard<std::mutex> lock(mtx);
        std::cout << "In critical section\n";
        // 自动解锁
    }
    
    // 方法内剩余的不涉及共享资源的操作
    ......    
}
```
